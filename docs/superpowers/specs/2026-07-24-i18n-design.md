# i18n 国际化设计

## 目标

将项目中所有面向用户的硬编码字符串替换为 errorCode + MessageSource 驱动的本地化方案，支持中文（默认）和英文。API 响应同时返回程序可读的 errorCode 和本地化 message；SSE 推送仅传 errorCode key，由前端自行翻译。

## 语言

| 角色 | 语言 |
|------|------|
| 默认 | 中文（zh_CN） |
| 第二语言 | 英文（en） |

语言由请求的 `Accept-Language` 头决定。

## 翻译分工

| 通道 | 翻译方 | 原因 |
|------|--------|------|
| REST API（`ApiResponse.message`） | 后端 | 有 `Accept-Language` 头，每次请求可准确获取用户语言 |
| SSE 推送（`WorkplaceStatusPayload.lockReasons`） | 前端 | SSE 是长连接，切换语言需要即时生效，重订阅不现实 |
| 持久化字段（`TaskRecord.faultMessage`、`export_task` 错误信息） | 存 errorCode | 存"发生了什么"这个事实，不存当时的翻译文案；前端展示时翻译 |

## ApiResponse 结构

```java
public record ApiResponse<T>(
    int code,                     // HTTP 状态码，不再永远是 200
    @Nullable String errorCode,   // 成功时为 null，失败时如 "task.name.duplicate"
    String message,               // 本地化文本（后端翻译）
    T data
) {}
```

**示例**：

```json
// 成功
{"code": 200, "errorCode": null, "message": "操作成功", "data": {...}}

// 业务错误
{"code": 409, "errorCode": "task.name.duplicate", "message": "任务名称已存在", "data": null}
```

## 文件结构

```
src/main/resources/i18n/
├── messages.properties      # 默认中文
└── messages_en.properties   # 英文
```

## Spring 配置

```yaml
spring:
  messages:
    basename: i18n/messages
    encoding: UTF-8
  web:
    locale: zh_CN
```

- `AcceptHeaderLocaleResolver` 自动解析 `Accept-Language` 头，无需自定义 Bean
- 默认 locale 通过 `spring.web.locale` 设为 `zh_CN`

## Messages 工具类

`com.tightening.i18n.Messages` — 静态工具类，通过 `ApplicationContextAware` 持有 `MessageSource` 引用。

```java
public final class Messages {
    // 获取当前请求 Locale 的消息（从 RequestContextHolder 读取）
    public static String get(String key, Object... args);

    // 显式指定 Locale（后台任务等无请求上下文场景）
    public static String get(Locale locale, String key, Object... args);
}
```

- 当前 Locale 从 `RequestContextHolder` 获取，没有请求上下文时回退到默认 zh_CN
- `MessageSource` 引用在 `ApplicationContextAware` 初始化时注入

## BusinessException

`com.tightening.i18n.BusinessException` — 携带 errorCode 的业务异常，替换现有的 `IllegalArgumentException` 裸字符串。

```java
public class BusinessException extends RuntimeException {
    private final String errorCode;
    private final Object[] args;
    private final HttpStatus httpStatus;  // 默认 400

    public static BusinessException of(String code, Object... args);         // 400
    public static BusinessException notFound(String code, Object... args);   // 404
    public static BusinessException conflict(String code, Object... args);   // 409
}
```

## 全局异常处理

新增 `com.tightening.i18n.GlobalExceptionHandler` — `@ControllerAdvice` + `@ExceptionHandler`。

```java
@ControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<ApiResponse<Void>> handleBusiness(BusinessException ex) {
        String msg = Messages.get(ex.getErrorCode(), ex.getArgs());
        return ResponseEntity.status(ex.getHttpStatus())
                .body(new ApiResponse<>(ex.getHttpStatus().value(), ex.getErrorCode(), msg, null));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiResponse<Void>> handleUnknown(Exception ex) {
        // 兜底：500 + 日志记录
    }
}
```

## SSE 推送改造

### LockReason

`LockReason` 枚举移除 `displayName` 字段，只保留 `key`。`WorkplaceStatusPayload.lockReasons` 从 `Map<String, String>`（key → 中文名）改为 `Set<String>`（key 的集合）。

```json
// 旧
{"lockReasons": {"pSetSending": "程序号下发中", "adminConfirm": "需管理员确认"}}

// 新
{"lockReasons": ["pSetSending", "adminConfirm"]}
```

前端拿到 key 后在本地翻译表查找 displayName。切换语言即时生效，无需重订阅 SSE。

对应的 errorCode key 为 `lock.pSetSending`、`lock.arrangerPositioning` 等（见上方 errorCode 列表），与 LockReason.key 的关系是：errorCode = `"lock." + LockReason.key`。实际使用时，`Messages.get("lock." + reason.getKey())` 获取翻译文本（仅 API 场景使用，SSE 不调 Messages）。

### WorkplaceStatus

`WorkplaceStatus` 枚举值当前通过 SSE 的 type 字段区分（`WORKPLACE_STATUS`），payload 中的 `status` 是枚举名（`UNACTIVATED` / `ACTIVATED` / `OPERATION_ENABLE` / `OPERATION_DISABLE`）。前端已基于枚举名展示，无需改动。

## 持久化字段的 errorCode 格式

以下持久化字段存储 errorCode 而非翻译后的文案：

| 字段 | 所属表 | 格式 | 示例 |
|------|--------|------|------|
| `faultMessage` | `task_record` | `errorCode` 或 `errorCode:arg1,arg2` | `task.circular_dependency` 或 `task.circular_dependency:5` |
| `error_message` | `export_task` | 同上 | `capability.failed:ExportData` |

**格式规则**：
- 冒号前为 errorCode，冒号后为逗号分隔的参数列表
- 无参数时只有 errorCode，不追加冒号
- 参数按原始类型存储（数字存数字、字符串存字符串），前端按需格式化
- 如果原始消息无法映射到任何 errorCode（如未预料的系统异常），存原始英文异常类名 `RuntimeException:Connection refused`

## ApiResponse 工厂方法

保留现有 `ok()` / `fail()` 签名，调整实现：

```java
public static <T> ApiResponse<T> ok(T data) {
    return new ApiResponse<>(200, null, Messages.get("common.ok"), data);
}

// 保留供 Controller 手动构造的场景使用
public static <T> ApiResponse<T> fail(HttpStatus status, String errorCode, Object... args) {
    return new ApiResponse<>(status.value(), errorCode, Messages.get(errorCode, args), null);
}
```

## 迁移范围

### 纳入迁移

1. **Controller** 中所有 `ApiResponse.fail("...")` → 抛 `BusinessException`，由 `@ControllerAdvice` 统一处理
2. **Service 校验** 中所有 `throw new IllegalArgumentException("中文消息")` → `throw BusinessException(...)`
3. **`BarcodeValidationResult`** 中的硬编码字符串 → 改用 errorCode
4. **`ApiResponse.ok()`** 中的 `"ok"` → `Messages.get("common.ok")`
5. **`LockReason`** 移除 `displayName`，`WorkplaceStatusPayload.lockReasons` 改为 key 集合
6. **`TaskRecord.markFaulted`** → 接收 errorCode 格式字符串
7. **`ExportResult.fail`** 和导出错误消息 → 改用 errorCode

### 不纳入本次

- 日志消息（`log.info/error`）— 日志本身不需要 i18n
- `AtlasCommandType` 等常量描述 — 内部使用，不暴露给前端
